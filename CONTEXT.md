# Domain Glossary — Open Wiki Studio

Names for the seams in this codebase. Architecture reviews and ADRs should
use these terms. Add a term here whenever a deepened module is named after a
concept not already listed.

## The wiki

**Wiki** — the `wiki/` folder inside a workspace; the agent-authored knowledge
base. A flat tree of Markdown files.

**Concept** — one Markdown file in the wiki, representing a single knowledge
unit. Identified by its **conceptId**.

**conceptId** — the stable identifier of a concept: the file's POSIX path
relative to `wiki/`, with the `.md` extension stripped. `wiki/foo/bar.md` →
`foo/bar`. Link targets and concept paths normalize to conceptIds through one
shared rule (strip an optional `wiki/` prefix and the `.md` suffix).

**Reserved file** — `index.md` and `log.md`. Generated by the wiki extension,
not agent-authored concepts. Whether a reserved file *counts* as a concept is
a per-caller policy (see `kind`), but the knowledge of which files are
reserved lives in one place.

**kind** — classification of a wiki Markdown file: `"concept"`, `"index"`, or
`"log"`. Lets each caller apply its own include/exclude policy without
re-deriving the reserved set.

## ConceptStore (the deepened module)

**ConceptStore** — the module that owns concept identity, metadata, and body.
Its interface is the test surface for everything concept-shaped:

- `listConcepts()` — every concept in the wiki, excluding reserved files
  (`kind === "concept"`).
- `listAll()` — every wiki Markdown file, including `index.md`/`log.md` tagged
  with their `kind`.
- `readConcept(relativePath)` — one concept by path, or `null`.
- `normalizeRef(ref)` — normalize a link/path reference to a conceptId (or
  `null` for external/non-concept refs). Shared with the graph's link
  extraction so the `wiki/` + `.md` stripping rule is not duplicated.

The store absorbs: recursive walk of `wiki/`, conceptId derivation, reserved
knowledge, and frontmatter parsing. Callers thin to their actual unique work:
`wiki-scan` hashes `body` and diffs; `wiki-graph` extracts links (its own
policy) and builds edges; `files.ts` previews `body` + metadata and keeps
only the non-wiki/text fallback.

The `Concept` record a store returns: `{ conceptId, kind, title, type,
description, tags, body }`. `type` is `concept.untyped` for concepts without
frontmatter `type`; `index`/`log` keep `type` as `concept.untyped` and let the
graph caller localize them (graph-specific vocabulary stays out of the store).

## Other seams (for reference)

**AgentRepository** — hosts the embedded Pi agent for one workspace; owns
the chat session pool, ingest orchestration, LLM configuration, and Copilot
OAuth. (See candidates C2–C4 in the architecture review for deepening
opportunities within it.)

**Ingest** — a `/wiki-update` run: snapshot the wiki, let the agent write
concepts + archive inputs, snapshot again, diff. The ingest summary
(created/updated/leftover) is computed from the before/after snapshots.

**Session** — a Pi chat session file; one or more live in the
AgentRepository's pool so background turns keep streaming.
